# Cosmos Checker — Security Model

## Обзор

Cosmos Checker работает с высокочувствительными данными (seed-фразы, приватные ключи). Несмотря на то что приложение read-only и не подписывает транзакции, утечка ключевого материала = полная потеря средств пользователя. Безопасность — критический приоритет.

## Threat Model

### T1: Утечка seed-фраз / приватных ключей с диска
- **Вероятность:** Высокая (если не обработать)
- **Импакт:** Критический — полная потеря средств
- **Митигация:**
  - Seed-фразы и приватные ключи НИКОГДА не записываются на диск
  - Хранятся ТОЛЬКО в оперативной памяти (RAM) на время сессии
  - После завершения проверки — зануление памяти (`zeroize` крейт в Rust)
  - В SQLite хранятся только bech32-адреса и результаты, НЕ ключевой материал
  - txt-файл с кошельками читается в память и не копируется

### T2: Утечка ключей через swap / crash dump
- **Вероятность:** Средняя
- **Импакт:** Критический
- **Митигация:**
  - Использование `zeroize` + `secrecy` крейтов — автоматическое зануление при Drop
  - Sensitive данные помечены как `Secret<T>` — нет случайного логирования/Debug вывода
  - Рекомендация пользователю: не использовать на машинах с неконтролируемым swap

### T3: Man-in-the-Middle при обращении к RPC нодам
- **Вероятность:** Средняя (публичные сети, прокси)
- **Импакт:** Низкий (read-only, нет подписи TX; максимум — подмена балансов)
- **Митигация:**
  - Использовать HTTPS/TLS для REST-эндпоинтов
  - Использовать TLS для gRPC (tonic с TLS)
  - Валидация TLS-сертификатов (НЕ отключать verify)
  - Предупреждение пользователю при использовании HTTP (без S) прокси

### T4: Вредоносный прокси-сервер
- **Вероятность:** Средняя
- **Импакт:** Низкий-Средний (подмена ответов API, сбор IP адресов)
- **Митигация:**
  - Через прокси НЕ передаются seed-фразы или ключи — только bech32-адреса
  - Адреса являются публичной информацией — их утечка некритична
  - Опционально: сравнение ответов от разных эндпоинтов для верификации

### T5: Утечка данных через логи
- **Вероятность:** Средняя (при отладке)
- **Импакт:** Критический (если логируются ключи)
- **Митигация:**
  - Тип `Secret<T>` из `secrecy` — при Debug/Display выводит `[REDACTED]`
  - Трейсинг (`tracing` крейт) с фильтрами: НИКОГДА не логировать поля с ключами
  - Код-ревью: grep по `seed`, `mnemonic`, `private_key` в логах

### T6: SQL Injection через данные Chain Registry
- **Вероятность:** Низкая
- **Импакт:** Средний
- **Митигация:**
  - Все SQL-запросы через параметризованные prepared statements (`rusqlite`)
  - Никогда не формировать SQL через string concatenation

### T7: Denial of Service от RPC нод
- **Вероятность:** Высокая (rate limits, нестабильные ноды)
- **Импакт:** Низкий (приложение замедляется, но не ломается)
- **Митигация:**
  - Таймаут на каждый запрос (5 сек по умолчанию)
  - Fallback chain: gRPC → REST → cosmos.directory
  - Round-robin по нескольким эндпоинтам
  - Semaphore для ограничения concurrency
  - Exponential backoff при rate limit (HTTP 429)

### Угрозы, НЕАКТУАЛЬНЫЕ для этого проекта

| Угроза | Почему неактуальна |
|--------|-------------------|
| XSS / CSRF | Нет web-сервера, Tauri изолирует WebView |
| Remote Code Execution через API | Нет внешнего HTTP API, только Tauri IPC |
| Подпись вредоносных TX | Приложение read-only, нет signing функций |
| Фишинг через UI | Десктоп-приложение, нет URL-бара |
| DDoS на приложение | Локальное приложение, нет открытых портов |

## Схема авторизации

Авторизация не требуется — это локальное десктоп-приложение без аккаунтов и без сервера. Вся логика выполняется на машине пользователя.

## Стратегия хранения секретов

| Тип данных | Где хранится | Как защищён |
|------------|-------------|-------------|
| Seed-фразы | Только RAM | `zeroize` + `secrecy`, зануление при Drop |
| Приватные ключи | Только RAM | `zeroize` + `secrecy`, зануление при Drop |
| Bech32-адреса | RAM + SQLite | Публичная информация, не секрет |
| Прокси (с паролями) | Только RAM | Зануление при завершении сессии |
| Результаты проверки | SQLite | Не содержат секретов |
| Настройки | SQLite | Не содержат секретов |

## Рекомендации по реализации

1. **Крейт `zeroize`** — обязателен для всех структур с ключевым материалом
2. **Крейт `secrecy`** — обёртка Secret<T> для предотвращения случайного логирования
3. **TLS everywhere** — не отключать верификацию сертификатов
4. **Prepared statements** — для всех SQL-запросов
5. **cargo audit** — запускать в CI перед каждым релизом
6. **Минимальные Tauri permissions** — в `tauri.conf.json` разрешить только необходимые API
